## 2. Development Goals
- Core objective: Build a menu-driven, object-oriented Pharmacy Management System that manages patients, doctors, medications, and prescriptions with clear class responsibilities and maintainable code organization.
- Required assignment features:
  - Add, delete, and edit patients, doctors, and medications.
  - Search patients, doctors, and medications by name.
  - Accept prescriptions by linking doctor, patient, and medication.
  - Add a patient to a doctor's managed patient list.
  - Generate a full system report of doctors, patients, medications, and prescriptions.
  - Check for expired medications and report them.
  - Print all prescriptions issued by a specific doctor.
  - Generate a past-year prescription summary showing medication names.
  - Restock medications across the pharmacy.
  - Provide a Scanner-based menu interface for team workflow.
  - Provide user and development documentation for usage, design, and build process.
- Non-goals/out of scope:
  - Persistent storage with a real database.
  - Authentication/authorization and user account roles.
  - GUI or web interface (console menu only for this submission).
  - Automated test suite (manual verification used for this iteration).

## 4. Class Responsibilities
### 4.1 Domain Model
- Person: Base class for shared identity and contact data (id, name, age, phoneNumber). Parent class for Patient and Doctor.
- Patient: Extends Person and stores patient-specific lists for medications and prescriptions. Supports add/remove operations for those lists.
- Doctor: Extends Person, stores specialization, and manages a list of assigned patients.
- Medication: Stores medication identity and inventory data (id, name, dosage, quantity, expirationDate). Expiry date is generated randomly in a mixed past/future range to support expiry-report testing.
- Prescription: Represents a doctor-issued prescription for one patient and one medication, with a default expiry of one year from issue date.

### 4.2 System and Workflow
- MedicationTrackingSystem: Central in-memory repository and CRUD/search service for patients, doctors, medications, and prescriptions. Also handles restocking logic.
- PrescriptionService: Validates prescription acceptance rules (entity registration, expiry, stock), links records, and decrements medication stock.
- PrescriptionReports: Generates formatted reports including full system report, prescriptions by doctor, and past-year medication summary.
- PharmacyMenu: Scanner-driven command loop that exposes all assignment features through numbered menu options with input validation.
- PharmacyManagementApp: Application entry point; creates system/services, seeds sample data, and starts the interactive menu.

## 5. Architecture and Relationships
- Object relationships and data flow:
  - User interacts with PharmacyMenu.
  - Menu calls MedicationTrackingSystem for CRUD/search/restock operations.
  - Menu calls PrescriptionService for prescription acceptance workflow.
  - Menu calls PrescriptionReports for generated report output.
  - Model objects (Patient, Doctor, Medication, Prescription) hold state and associations used by services.
- Key OOP principles used:
  - Inheritance: Patient and Doctor inherit from Person.
  - Encapsulation: Fields are private, with controlled access through getters/setters and service methods.
  - Composition/association: MedicationTrackingSystem composes collections of all core entities.
  - Separation of concerns: Menu handles input/output, services handle business logic, models handle data.
- Association summary:
  - Doctor -> many Patients
  - Patient -> many Prescriptions
  - Patient -> many Medications
  - Prescription -> one Doctor, one Patient, one Medication
  - MedicationTrackingSystem -> lists of all Patients, Doctors, Medications, Prescriptions

## 6. Build and Run Process
### 6.1 Compiler and Runtime
- Java version: Java 17 or newer recommended.
- JDK/JRE details: Requires JDK for compile and JRE/JDK for run.

### 6.2 Build Commands

- Windows PowerShell:


## 10. Feature Implementation Notes
### 10.1 CRUD operations
- Add/delete/edit/search behavior and where implemented:
  - Primary menu actions are implemented in PharmacyMenu.
  - Data mutations and lookups are implemented in MedicationTrackingSystem:
    - addPatient/addDoctor/addMedication/addPrescription
    - removePatient/removeDoctor/removeMedication/removePrescription
    - editPatient/editDoctor/editMedication
    - findPatient/findDoctor/findMedication/findPrescriptions
  - Search is case-insensitive by name.

### 10.2 Prescription acceptance flow
- Validation steps:
  - Verify system and prescription are non-null.
  - Verify doctor, patient, and medication exist and are registered.
  - Verify medication is not expired.
  - Verify stock quantity is greater than zero.
- Data linking behavior:
  - On success, link patient to doctor if needed.
  - Add prescription to patient list and system list.
  - Add medication to patient list if missing.
- Stock update behavior:
  - Decrement medication quantity by 1 per accepted prescription.
  - Quantity is clamped to zero minimum.

### 10.3 Reporting
- Full system report:
  - Implemented in PrescriptionReports.generateFullSystemReport.
  - Prints doctors, patients, medications, and prescriptions.
- Expired medications report:
  - Implemented in PharmacyMenu option 8.
  - Filters medications where expirationDate is before current date.
- Doctor prescription report:
  - Implemented in PrescriptionReports.prescriptionByDoctor.
  - Prints prescriptions for the selected doctor.
- Past-year patient medication summary:
  - Implemented in PrescriptionReports.patientPrescriptionPastYear.
  - Aggregates and prints medication names with counts for the past year.

### 10.4 Restocking
- Rules and update logic:
  - Implemented in MedicationTrackingSystem.restockAllMedications.
  - Positive restock amount is required.
  - Adds restock amount to every medication quantity.
  - Resets expiration date to one year from current date for restocked medications.


## 11. Theoretical Database Design
### 11.1 Why a database might be used
- Persistence goals:
  - Persist doctors, patients, medications, and prescriptions across application restarts.
  - Improve data integrity with relational constraints.
  - Support future reporting and analytics with queryable historical data.

### 11.2 Proposed entities/tables
- person
- patient
- doctor
- medication
- prescription
- doctor_patient

### 11.3 Table fields and keys
- Primary keys:
  - person.person_id
  - medication.medication_id
  - prescription.prescription_id
  - patient.patient_id
  - doctor.doctor_id
- Foreign keys:
  - patient.patient_id -> person.person_id
  - doctor.doctor_id -> person.person_id
  - prescription.patient_id -> patient.patient_id
  - prescription.doctor_id -> doctor.doctor_id
  - prescription.medication_id -> medication.medication_id
  - doctor_patient.doctor_id -> doctor.doctor_id
  - doctor_patient.patient_id -> patient.patient_id
- Key constraints:
  - person.person_id unique and not null.
  - prescription references valid doctor/patient/medication rows.
  - medication.quantity >= 0 check constraint.
  - prescription.expires date not null.
  - doctor_patient has composite primary key (doctor_id, patient_id).

### 11.4 Entity Relationships
- One-to-many and many-to-many mappings:
  - One patient can have many prescriptions.
  - One doctor can issue many prescriptions.
  - One medication can appear in many prescriptions.
  - Doctor-to-patient is many-to-many through `doctor_patient(doctor_id, patient_id)`.
- Relationship notes:
  - `person` is a shared base table for common attributes.
  - `patient` and `doctor` act as role-extension tables of `person`.
  - `prescription` is the central transactional table connecting all core entities.
  - `doctor_patient` supports direct doctor-patient assignment behavior used in the menu flow.
